## Context

当前系统已经具备章节融合能力，能够生成 `FusionDraft` 并在工作台中查看结果，但发布链路缺少一层正式的、一致性的准入校验。PRD 要求系统在拿到 `FusionDraft` 或 `MergedDraft` 后，联合 `ChapterPlan`、`StateLocks`、`FactStore` 和 `StoryBible` 执行三级检测，并在存在未解决 P0 时直接阻断发布。

这次变更是一个跨后端服务、数据模型、前端工作台和发布流程的改动，且包含需要 LLM 参与的语义判定，因此需要在编码前固定责任边界和执行顺序。

## Goals / Non-Goals

**Goals:**
- 提供统一的章节校验入口，对单个草稿生成结构化 `ValidationReport`
- 将校验拆分为规则检测、状态比对、语义判定三个可组合阶段，保证确定性问题优先被捕获
- 为 P0、P1、P2 问题建立统一的数据模型，支持定位片段、处理状态和补丁建议
- 将 P0 未解决状态接入发布前门禁，前端必须禁用发布动作并展示原因
- 在章节工作台与校验中心中展示问题列表，并提供“生成修复补丁”入口

**Non-Goals:**
- 不在本次变更中实现完整自动修稿闭环，只预留补丁生成入口和返回结构
- 不尝试把所有编辑质量问题都定义成 P0，风格类或弱语义类问题仍留给 P1/P2
- 不改造草稿生成算法本身，校验器只消费现有 `FusionDraft` / `MergedDraft` 与上下文状态

## Decisions

### 1. 采用同步触发、异步执行、可轮询读取结果的校验模型

`POST /api/chapters/{chapterId}/validate` 负责创建一次校验任务并返回报告摘要；服务端允许先进入执行态，再在完成后落盘完整报告。这样可以兼容规则检测和 LLM 语义判定耗时不一致的情况，同时保持接口简单。

选择该方案而不是“请求内同步完成所有校验”，是因为语义判定和补丁建议在章节较长时会显著拉长响应时间，并影响工作台交互体验。

### 2. 三层校验共用同一份标准化 issue 模型

所有校验器都输出统一的 `ValidationIssue` 结构：
- `severity` (`p0` / `p1` / `p2`)
- `code`
- `title`
- `message`
- `spans`
- `blocking`
- `suggest_patch`

这样前端无需感知问题来源，后续也可以增加新的规则或新的 LLM 提示模板，而不影响展示和发布门禁逻辑。

替代方案是为每类校验器定义独立返回结构，再在聚合层映射。该方案扩展成本高，也会把前端和存储层耦合到校验实现细节，因此不采用。

### 3. `FusionDraft` 与 `MergedDraft` 共用统一 span 规则

两类草稿共用同一套段落定位规则与 `spans` 结构，校验、Diff、Patch 和前端高亮都基于统一的 paragraph index 语义工作。只要两类草稿仍然保持相近的段落化文本结构，就不引入双解析器。

替代方案是按草稿类型维护两套 span 解析器。该方案会把校验与前端展示逻辑拆成两套口径，增加错位和回归风险，因此不采用。

### 4. 规则层与状态层必须完全确定性，语义层仅处理无法静态判定的问题

规则检测负责捕获可直接从文本或结构中识别的问题，例如互斥事实并存、重复终态片段、未闭合引用等；状态比对负责校验草稿终态与 `StateLocks` / `ChapterPlan` 的一致性。只有当问题依赖语义别名、人物身份映射、模糊表述归并时，才进入 LLM 判定。

这样做可以降低 LLM 成本和不稳定性，并保证明显错误不被模型误判掩盖。

### 5. 为语义判定定义最小 LLM 网关与预算约束

本次变更需要一并定义 LLM 语义判定最小网关策略，至少包括：
- 仅身份漂移、别名映射冲突、模糊终态归并失败等无法由规则层确定的问题允许调用 LLM
- 对单章语义判定消耗的 input/output token 做累计记录，并作为预算控制与前端展示依据

这样可以在第一版就锁住成本和时延边界，避免后续校验链路失控。

### 6. 发布门禁以“存在未解决 blocking issue”为准，而不是仅看最新报告 passed 字段

发布按钮和发布接口都应检查章节当前关联草稿是否存在最新一份、与当前草稿版本一致的报告，并确认没有未解决的 `blocking=true` 问题。单纯依赖 `passed` 布尔值不足以覆盖“旧报告已过期”或“问题被人工标记后重新生成了新报告”的情况。

因此报告模型需要记录 `draft_type`、`draft_id`、输入版本号，以及问题级别和处理状态。发布前必须校验报告与当前草稿上下文匹配。

### 7. P0 问题不允许通过“忽略”绕过发布门禁

`ignored` 状态只适用于 P1/P2。P0 作为阻断级问题，不能通过“已忽略”直接放行发布；若存在误报，需要进入人工复核或白名单机制，但不属于本次变更范围。

这样可以保持 P0 的含义稳定，避免团队后续把阻断问题退化成形式检查。

### 8. 持久化采用报告表 + issue 明细表，两者保持一对多

`validation_reports` 保存摘要、版本和时间戳，`validation_issues` 保存具体问题、定位和处理状态。这符合 PRD，且支持校验中心跨章节筛选与后续审计。

不采用将完整问题 JSON 全部塞入单表的做法，因为问题筛选、状态流转和运营查询都需要单 issue 级别索引。

### 9. 校验触发采用“保存后自动触发 + 发布前强制触发”

章节草稿保存后自动触发一次校验，用于尽早暴露问题并更新工作台状态；发布前无论是否已有报告，都必须基于当前草稿与输入版本再强制触发一次，作为最终守门。

不采用“仅手动触发”的方案，因为这会让校验脱离主流程，无法稳定承担发布门禁职责。

### 10. “生成修复补丁”入口采用惰性生成

对于 `suggest_patch=true` 的问题，前端仅展示按钮，不在主校验流程中强制生成补丁。补丁建议可复用语义判定的上下文与 issue 描述，但应由单独动作触发，避免校验主链路被额外 LLM 调用拖慢。

## Risks / Trade-offs

- [LLM 语义判定结果不稳定] → 先用规则层与状态层兜底；语义层输出需要包含置信度或可审阅说明，低置信结果降级到 P1
- [LLM 成本或时延失控] → 对单章 token 消耗做可观测记录，并把消耗记录暴露给前端，便于后续调参与治理
- [报告与草稿版本错配导致误阻断或漏阻断] → 报告写入时保存 `draft_id`、`plan_version`、`state_lock_version`，发布前必须做版本一致性检查
- [校验耗时过长影响工作台体验] → 接口返回摘要状态，详情页轮询或刷新读取完整报告；补丁生成从主链路拆出
- [P0 判定范围过宽导致发布频繁受阻] → 将 P0 严格限定为事实冲突、终态冲突、关键身份漂移等会破坏剧情连续性的错误，其他问题先落 P1/P2
- [前端状态分散，多个页面展示口径不一致] → 全部页面以统一报告接口与统一 issue 模型渲染，避免页面自行拼接逻辑

## Migration Plan

1. 新增校验报告与问题表，并补充必要索引
2. 在章节服务中接入校验创建和查询接口
3. 实现三层校验流水线与统一 span 解析，先上线规则层和状态层，再接入带 token 消耗记录的语义层
4. 在章节工作台右侧增加 Validation 面板，并接入报告详情展示与单章 LLM token 消耗展示
5. 在草稿保存后接入自动校验触发
6. 在发布确认弹窗和后端发布接口中接入发布前强制校验与 blocking issue 检查
7. 新增校验中心页面，支持跨章节筛选和处理状态流转，其中 P0 不允许被忽略
8. 补丁生成功能可在校验主链路稳定后灰度开放

回滚策略：
- 保留数据表但关闭发布门禁开关，前端仍可只读展示历史报告
- 若语义层异常，可单独关闭 LLM 判定，仅保留规则层和状态层

## Open Questions

- P0 误报进入人工复核或白名单时，复核结果由哪个角色写入，以及是否需要单独审计字段
- 自动触发校验的去抖策略是否需要按保存频率做节流，以避免连续保存触发过多后台任务
