## Context

当前已归档的 `chapter-fusion` change 把 `StateLocks` 视为融合输入之一，也要求记录 `state_lock_version`，但系统还没有一条正式的“先生成锁、再开始融合”的链路。结果是融合能力在实现层面容易退化成“按章节取最新上下文就开始写”，把事实边界漂移、人物禁入、终态偏差等问题留给后置校验或人工发现。

用户这次要求不是补一个独立接口，而是修正整条章节生产顺序：
1. 先生成并冻结章节状态锁
2. 融合器在同一份锁约束下生成整章
3. P0 阻断式校验在同一份锁语义上守门

因此本次设计要同时覆盖后端状态锁生成、版本化保存、融合前置校验、工作台展示与和 `add-p0-blocking-validation` 的输入契约对齐。

## Goals / Non-Goals

**Goals:**
- 在章节进入 BeatDraft / Fusion / Validation 之前，自动生成首版 `StateLocks`
- 将状态锁定义为可版本化、可人工微调、可审计的章节事实边界
- 让 BeatDraft、Fusion、Validation 在同一章节流程内读取同一 `state_lock_version`
- 要求 Fusion 只能消费明确存在的 `state_lock_version`，不允许隐式使用“当前最新状态”
- 为 Validation/Patch/Merge 提供同一份锁版本引用，确保下游检查对象一致
- 在章节工作台提供状态锁查看、编辑、冲突提示能力

**Non-Goals:**
- 不在本次 change 内完整实现 `add-p0-blocking-validation` 中的所有阻断规则
- 不重新设计 BeatDraft 生成策略，只要求其输入契约绑定状态锁版本
- 不尝试自动解决所有锁冲突，本次只定义生成、版本、消费和提示/阻断边界

## Decisions

### 1. 状态锁采用“章节主记录 + 版本快照”双表模型

新增两张表：
- `state_locks`：保存章节当前锁头信息与当前版本号
- `state_lock_versions`：保存每次生成或人工编辑后的完整快照

锁分组按 PRD 固定为 `time_lock`、`location_lock`、`character_lock`、`item_lock`、`numeric_lock`、`event_lock`、`ending_lock`。这样可以支持“获取当前锁”和“回溯历史版本”两个场景，同时保证人工修改有审计线索。

不采用“只有一个 JSON 字段覆盖写”的方案，因为后续 Fusion、Validation 和发布阻断都需要明确引用版本号，覆盖写无法稳定重放上下文。

### 2. 章节生成前通过专用接口创建首版状态锁

定义 `POST /api/chapters/{chapterId}/state-locks` 负责读取：
- 上章终态
- `ChapterPlan`
- `FactStore`
- `StoryBible`

服务在一个原子流程中产出首版锁快照，并返回 `state_lock_id`、`version` 和完整 `locks` 结构。若章节已有状态锁，接口默认生成新版本而不是静默复用旧结果，以便当 `plan_version` 或上游上下文变化时重新冻结边界。

不把状态锁生成塞进融合接口内部，因为这会继续维持“融合顺手补锁”的错误顺序，也不利于用户在融合前人工微调。

### 3. 所有下游阶段必须显式绑定同一 `state_lock_version`

从设计层面固定以下输入契约：
- `BeatDraft`
- `Fusion`
- `Validation`
- `Patch`
- `Merge`

这些阶段的记录或任务参数都必须保存 `state_lock_version`，并在读取时校验引用版本仍然存在。`BeatDraft`、`FusionDraft` 和后续 `Validation` 不允许依赖“章节当前最新锁”，而必须消费请求参数里的明确版本；若未提供版本或章节尚未生成锁，则直接阻断。

替代方案是统一取章节当前版本。该方案实现更快，但只要用户在 Beat 或融合后编辑过锁，就会导致 Beat、Fusion、Validation 读到不同事实边界，排查会非常困难，因此不采用。

### 4. 状态锁冲突分为“提示”和“阻断”两级，但发布阻断归 Validation 所有

本 change 负责两类反馈：
- 工作台内即时提示：在正文预览、融合结果或校验结果中标记“违反状态锁”
- 融合前置阻断：没有锁版本或锁版本无效时，不允许启动融合

至于“正文内容已经违反锁，是否进入 P0 阻断发布”，本次不重复定义全部规则，而是与 `add-p0-blocking-validation` 对齐：Validation 必须读取与草稿相同的锁版本，对数值锁、人物禁入、终态锁等冲突给出结构化问题，并在 P0 情况下阻断发布。

这样拆分可以避免两个 change 重复拥有“谁来最终守门”的职责。

### 5. 自动生成逻辑按“硬边界优先，开放项显式留空”执行

自动生成首版状态锁时：
- 可从上章终态、Plan 或 FactStore 明确推出的约束，直接写入锁
- 无法稳定确定的字段，不做强推断，留空或写入开放集
- 终态落点、禁入人物、关键数值等高风险项优先落锁

这样可以把锁定义成“边界合同”，而不是伪装成全知剧情摘要。过度推断会放大误锁风险，并把后续生成压到错误边界内。

### 6. 仅在必要的歧义判断环节引入 LLM 辅助

状态锁生成默认优先使用确定性来源与规则归纳，但以下场景允许引入 LLM：
- 人物别名、称谓和身份映射无法稳定通过字典规则归并
- 章节终态表述模糊，需要从多段上下文归纳单一落点
- 事件或道具状态存在隐含表达，规则层无法稳定抽取

LLM 在这些环节的职责是输出“结构化锁候选”或“待确认建议”，而不是直接替代锁版本本身。服务端仍需把最终结果落成结构化 `StateLocks` 快照，并与版本号绑定，供 Fusion 和 Validation 消费。

不采用“整份状态锁完全交给 LLM 生成且不做结构约束”的方案，因为这会削弱锁作为系统边界合同的作用，也会让后续校验失去稳定输入。

### 7. 状态锁变更需要显式传播到已生成草稿

当新的状态锁版本与已生成草稿绑定的旧版本相比，影响到以下任一关键域时：
- `ending_lock`
- `numeric_lock` 中的关键数值
- `character_lock` 中的出场/禁入人物
- `event_lock` 中的禁止事件

系统需要自动把相关 `FusionDraft` 标记为 `stale` 或 `needs_refusion`。这样前端不会把已经脱离当前边界的融合稿继续当作可信结果展示。

不采用“只更新锁版本，不回写草稿状态”的方案，因为这会让工作台保留看似可用、实际已经失去可信度的 FusionDraft。

### 8. 状态锁生成支持按指定 plan version 重建

状态锁生成接口需要支持最小重建参数集：
- `chapter_id`
- `plan_version`
- 是否读取上章终态

这样历史回放、问题复现、回滚重跑都能在稳定输入上重建新的 `state_lock_version`。重建结果仍然写成新的锁版本，而不是覆盖历史版本。

不采用“只能基于当前最新 Plan 生成”的方案，因为这会让问题复现和历史对账失去可重复性。

### 9. 前端采用章节工作台右栏卡片 + 编辑弹窗 + 冲突提示三件套

工作台右栏新增 State Locks 卡片，按锁分组展示，并给每条锁增加三种状态：
- 正常
- 被正文违反
- 已被人工修改

编辑入口使用弹窗，强制填写修改原因并触发新版本生成。正文或报告中的冲突提示需要能跳转回具体锁项，保证用户知道是“改文本”还是“改锁”。

不新建独立页面作为主入口，因为锁的使用语境天然贴近章节工作台与融合预览。

## Risks / Trade-offs

- [自动生成的锁过严，误伤后续创作] → 只对可确定事实落硬锁；高不确定项保持开放并允许人工微调
- [LLM 辅助生成带来不稳定候选] → 仅在规则难以覆盖的歧义环节调用；输出先归一为结构化候选，再由版本化保存和人工微调兜底
- [状态锁更新后旧 FusionDraft 继续被误用] → 对关键锁域做差异检测，命中后自动标记 `stale` / `needs_refusion`
- [用户在融合后编辑锁，导致旧草稿与新锁错配] → 所有草稿与任务都保存 `state_lock_version`；展示层同时提示“当前锁版本已更新”
- [锁结构过于松散，前端难以稳定渲染] → 固定一级锁分组和常见字段命名，扩展字段放入对应 JSON 内但不改变分组骨架
- [职责重叠，状态锁提示和 P0 校验重复判断] → 本 change 只定义锁生成、版本和消费契约；最终发布阻断继续由 Validation change 负责
- [没有显式迁移顺序，开发时又回到先做融合再补锁] → 任务拆分中把“状态锁生成与版本绑定”放在融合入口改造之前

## Migration Plan

1. 新增 `state_locks`、`state_lock_versions` 表与基础查询索引
2. 实现状态锁生成接口、当前锁查询接口、编辑接口与版本快照写入
3. 改造章节工作台，接入锁卡片、编辑弹窗和冲突提示占位
4. 改造 Beat/Fusion 入口，要求显式传入并保存 `state_lock_version`
5. 与 `add-p0-blocking-validation` 对齐报告输入，确保 Validation 读取草稿所绑定的同一锁版本
6. 为状态锁版本变更增加关键域差异检测，并把受影响的 FusionDraft 标记为 `stale` / `needs_refusion`
7. 上线时先开启“先生成锁再融合”的链路约束，再让发布校验切到共享锁版本

回滚策略：
- 保留状态锁数据表，但关闭前端入口和融合前置阻断开关
- 若自动生成质量不稳定，可先保留人工创建/编辑能力，暂停自动生成入口

## Open Questions

- `numeric_lock` 与 `event_lock` 中哪些字段应被视为“关键变更”，从而触发 `needs_refusion`
- BeatDraft 在锁版本变化后是直接标记失效，还是允许保留历史只读查看
