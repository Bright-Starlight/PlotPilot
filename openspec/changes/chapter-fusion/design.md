## Context

现有流程已经能产出 ChapterPlan 和 BeatDraft，但缺少一个专门的章节级收束层来处理重复信息、过桥、终态收敛和发布前校验。这个 change 要补的是“从节拍到整章”的编排层，而不是再扩一套新的创作模式。

## Goals / Non-Goals

**Goals:**
- 将多个 BeatDraft 稳定融合为单个可校验的 FusionDraft。
- 保留来源信息、重复率、确认事实、开放问题、终态和警告，便于验证与回溯。
- 提供融合任务创建、状态查询、融合预览和章节工作台查看入口。
- 让融合失败、阻断和 warning 的边界清晰可追踪。

**Non-Goals:**
- 不新增新的创作策略或额外的 AI 写作模式。
- 不改变 Beat 层的规划方法。
- 不在本次 change 中扩展发布链路之外的其他章节生产能力。

## Decisions

1. 采用独立的融合任务模型，而不是把融合直接塞进 Beat 生成流程。
   - 原因：融合有自己的输入校验、失败状态、日志和重试需求，独立任务更容易追踪和恢复。
   - 备选方案：在 Beat 生成后同步执行融合。该方案简单，但会把长耗时和失败分支耦合到上游流程。

2. 将 `chapter_fusion_drafts` 与 `fusion_job_logs` 作为独立表保存。
   - 原因：融合结果需要版本化保存，日志需要按步骤追踪，不应混在章节主表里。
   - 备选方案：把结果写回章节表或通用日志表。这样会让审计、重跑和查询变复杂。

3. 用 `plan_version` 和 `state_lock_version` 作为融合输入的显式版本门。
   - 原因：融合必须对齐可重复的上下文，避免同一章节在不同锁状态下生成不一致结果。
   - 备选方案：仅按 chapterId 拉最新数据。该方案实现更快，但不可重放，也容易引入漂移。

4. 让重复去重、过桥补齐、终态统一和长度裁剪共享同一条融合管线。
   - 原因：这些步骤彼此影响，拆成互不关联的多个后处理阶段会增加重复逻辑和状态分叉。
   - 备选方案：为每一步建立独立异步任务。该方案更灵活，但会显著增加系统复杂度。

5. 前端采用章节工作台内的 Tab 和预览弹窗，而不是新建独立工作台页面。
   - 原因：用户语境已经在章节维度，入口应尽量贴近 Beat 和 Fusion 的切换。
   - 备选方案：独立 Fusion 页面。该方案更自由，但会让章节编辑链路割裂。

## Risks / Trade-offs

- [重复去重误判] → 用 warning 记录“可能重复”而不是静默删改，保留人工复核入口。
- [终态冲突] → 将终态唯一性作为硬约束；无法收敛时直接失败，避免发布错误章节。
- [长度裁剪损伤语义] → 先保留结构一致性，再做最小裁剪；必要时通过 warning 暴露风险。
- [preview 与最终结果不一致] → 预览复用同一套评分与校验数据源，减少双轨逻辑。
- [融合结果回溯困难] → 保存 source beat IDs、版本号和日志，保证可追查。

## Migration Plan

1. 新增融合任务接口和融合草稿存储。
2. 接入融合作业执行器，先支持 queued -> running -> completed/blocked/failed 的基础状态流。
3. 将章节工作台的 Beat/Fusion/Diff 视图接入新数据源。
4. 逐步把 validation 接到 FusionDraft 输出上，保留现有发布链路不变。
5. 回滚时关闭融合入口并停止调度，不删除已有融合草稿，避免数据丢失。

## Open Questions

- repeat ratio 的告警阈值由配置固定还是按章节类型区分。
- preview 是否需要异步刷新，还是提交后立即返回估算值即可。
- warning 与 blocked 的边界是否需要按章节阶段再细分。
